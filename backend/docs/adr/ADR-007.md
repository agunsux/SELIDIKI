# ADR-007: Firestore Decommission Criteria

## Status
Approved

## Context
Removing Firestore SDK dependencies, environment keys, and cleanup code is destructive and permanent. This cleanup must only be executed when PostgreSQL is fully verified to be stable and self-sustaining.

## Decision
Establish the following criteria that must be satisfied before decommissioning Firestore:
1. **Observation Period:** PostgreSQL must run as the primary production data store for at least 14 consecutive days with zero rollbacks.
2. **Zero Data Drift:** Comparison checks on historical records must verify 0% mismatch.
3. **Parity Maintenance:** API response, telemetry, and error rates must be stable.
4. **Disaster Recovery Validation:** Automated point-in-time recovery (PITR) and backups must be operational.
5. **No Performance Degradation:** Average API response latency must remain within the baseline (+/- 5%).
6. **Executive Sign-off:** Sign-off by engineering leadership following validation scorecard audit.

## Alternatives Considered
* **Immediate Decommissioning:** Decommissioning Firestore inside Sprint 2. Rejected because we lose the rollback safety net.

## Consequences
* **Enhanced Safety:** Ensures Postgres is stable under peak loads, weekly schedules, and background worker jobs before code cleanup.
* **Controlled Deletion:** Decommissioning is handled as a dedicated Sprint 3 task, keeping Sprint 2 focused on preparation.

## Risks
* Running both databases during the observation period increases infrastructure costs, which is accepted as a cost for safety.

## Rollback Implications
Once decommissioning is complete, rollback to Firestore is no longer possible.

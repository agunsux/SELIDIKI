# ADR-001: Repository Abstraction for Unified Data Access

## Status
Approved

## Context
The SELIDIKI backend is migrating its primary data store from Firestore to PostgreSQL. Historically, routes and middleware have directly imported database clients or used direct database queries. Continuing this approach during a multi-phase migration increases risk, duplicates vendor routing code, and complicates fallback mechanisms.

## Decision
Introduce a Repository Abstraction Layer. All controllers, middlewares, and services must interact exclusively with unified repository classes (e.g., `UserRepository`, `PhoneRepository`, `BankAccountRepository`, `ReportRepository`, `HistoryRepository`). Direct imports of `firebase-admin/firestore` or raw client queries using `utils/db` are strictly forbidden outside of the designated repository provider subfolders (`repositories/firestore/` and `repositories/postgres/`).

## Alternatives Considered
* **Inline Conditionals:** Routing between database queries inside routers/middleware using `if (provider === 'FIRESTORE')`. Rejected because it duplicates code and increases implementation drift.
* **Service-Level Routing:** Routing at the service layer. Rejected because the codebase has a lean structure where route handlers directly call repositories for data access, meaning service routing would add redundant boilerplate.

## Consequences
* **Decoupling:** Routers and authentication middleware remain completely unaware of the underlying database engine.
* **Testability:** Unit tests can mock the unified repository interfaces easily.
* **Safety:** Changes to Postgres or Firestore models are encapsulated entirely within their respective repository files.

## Risks
* Minor latency overhead from an additional call layer (negligible in Node.js/V8).
* Drift in repository class schemas if not validated by contract tests.

## Rollback Implications
Simplifies rollback. If a problem is discovered, database routing can be switched instantly via environment variables inside the repository factory layer.

## Future Evolution
Allows the backend to plug in caching layers (e.g., Redis) or other databases in the future without modifications to the router or controller contracts.

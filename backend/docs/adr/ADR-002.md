# ADR-002: Database Provider Strategy

## Status
Approved

## Context
A big-bang cutover from Firestore to PostgreSQL is high-risk, as rollback is difficult, parity is unproven under live conditions, and data loss could occur. A structured phased approach is necessary to ensure zero downtime.

## Decision
Adopt a phased transition strategy using database provider modes. We will configure 5 distinct states:
1. `FIRESTORE`: Firestore serves all reads/writes. Postgres is offline or untouched.
2. `DUAL_WRITE`: Firestore serves reads. Writes are sent synchronously to Firestore and duplicated (best-effort/safely) to PostgreSQL.
3. `DUAL_READ`: Both databases are read. Firestore is returned, and values are compared. Discrepancies are logged. Writes are still dual-written.
4. `SHADOW`: Replay live production read/write operations asynchronously to PostgreSQL.
5. `POSTGRES`: PostgreSQL serves all reads and writes. Firestore serves as an inactive standby.

## Alternatives Considered
* **Big-Bang Cutover:** Stop the server, migrate data, redirect config to Postgres, start the server. Rejected because any data model or query issues would cause severe production downtime, and rollback would require reverse-migrating newly written data.

## Consequences
* **Mitigated Downtime:** The transition can be executed online.
* **Telemetry and Validation:** We collect live comparisons before making Postgres the source of truth.
* **Immediate Rollback:** Standby Firestore remains up-to-date.

## Risks
* Double-writing increases API call overhead and database load.
* Requires database synchronization scripts to seed historical data.

## Rollback Implications
At any point before decommissioning, setting `DATABASE_PROVIDER=FIRESTORE` safely returns the system to the legacy baseline.

## Future Evolution
Once PostgreSQL is fully stable in production and decommissioning is approved, the provider routing layer can be cleaned up in Sprint 3.

# ADR-008: Internal Repository Firestore Adapters

## Status
Proposed (Generated during Phase 0 Architecture Audit)

## Context
The SELIDIKI backend has 9 repositories in production. As of Sprint 2A:
- **5 unified repositories** (Phone, BankAccount, Report, History, User) have both Firestore and PostgreSQL adapters.
- **4 internal repositories** (FraudEntity, FraudReport, LookupLog, Url) are PostgreSQL-only with no Firestore fallback.

These 4 internal repositories serve:
- `FraudEntityRepository.findByHash()` — Used by `reputationService.check()` (endpoint #13)
- `FraudReportRepository.findByEntityId()` — Used by `reputationService.check()` (endpoint #13)
- `FraudEntityRepository.ping()` / `FraudReportRepository.ping()` — Used by `reputationService.checkDatabaseConnection()` (endpoint #14)
- `LookupLogRepository.insert()` — Asynchronous audit logging
- `UrlRepository.findByDomain()` / `upsert()` — URL intelligence (no active route)

This creates a **critical rollback gap**: if PostgreSQL fails in production, reputation endpoints (#13, #14) cannot fall back to Firestore, violating ADR-002 (Database Provider Strategy) and ADR-006 (Rollback Strategy).

## Decision

**Option A (Recommended): Create Firestore adapters for all 4 internal repositories.**

Rationale:
- Completes the dual-DB coverage to 100%
- Enables full rollback capability for all 15 endpoints
- Satisfies the governance requirement that "every repository must have Firestore + Postgres implementations"
- Follows the established pattern from the 5 unified repositories

Implementation:
1. Create `repositories/firestore/FraudEntityRepository.js` — Map `phone_profiles`/`account_profiles` collections to `FraudEntity` domain model
2. Create `repositories/firestore/FraudReportRepository.js` — Map `fraud_reports` collection to `FraudReport` domain model
3. Create `repositories/firestore/LookupLogRepository.js` — Map to `audit_log` Firestore collection
4. Create `repositories/firestore/UrlRepository.js` — Map to `url_profiles` Firestore collection
5. Create unified abstractions for each that use `databaseProvider.js`
6. Update `reputationService.js` to use unified abstractions

## Alternatives Considered

**Option B: Formally exempt as secondary services.**
- Defer Firestore adapters to Sprint 2B or Sprint 3
- Accept that reputation endpoints (#13, #14) are PG-only during Sprint 2A
- Mark as acceptable risk since reputation is not a core user path
- **Rejected:** Violates the governance requirement for 100% repository coverage and creates a rollback gap that could be problematic if PostgreSQL issues arise.

**Option C: Merge reputation service into unified repositories.**
- Replace `FraudEntityRepository`/`FraudReportRepository` with queries through existing `PhoneRepository`/`ReportRepository`
- **Rejected:** Would overload unified repositories with domain logic, violates SRP, and creates circular dependencies.

## Consequences

- **Coverage:** Repository dual-DB coverage increases from 55.6% to 100%
- **Rollback Safety:** All 15 endpoints can fall back to Firestore
- **Consistency:** All repositories follow the same abstraction pattern
- **Effort:** ~4 new Firestore adapters + 4 unified wrappers + service refactor
- **Risk:** Low — Firestore schemas already exist for all underlying collections



## Legacy Service Cleanup (Required by Architecture Review)

As part of this ADR, the following legacy files MUST be removed or fully refactored:
- `services/fraudGraph.js` — Contains 4 direct `getFirestore()` calls. Functions: `getPhoneProfile`, `getAccountProfile`, `createFraudReport`, `getTrendingReports`. All are superseded by unified repositories.
- `services/historyService.js` — Contains 1 direct `getFirestore()` call. Function: `saveToHistory`. Superseded by `HistoryRepository.insert()`.

**Verification:** After implementation, run:
```bash
grep -r 'getFirestore' backend/ --include='*.js' | grep -v repositories/firestore/ | grep -v node_modules
```
Result must be **empty**. Any non-empty result is a BLOCKER.

## Dependency Injection Strategy (Required by Architecture Review)

This ADR adopts the **Static Method Pattern** for dependency resolution:

```javascript
// repositories/FraudEntityRepository.js (unified)
const FirestoreFraudEntityRepository = require('./firestore/FraudEntityRepository');
const PostgresFraudEntityRepository = require('./postgres/FraudEntityRepository');
const flags = require('../config/featureFlags');

class FraudEntityRepository {
  static async findByHash(hash) {
    if (flags.ENABLE_DUAL_READ) {
      const fs = await FirestoreFraudEntityRepository.findByHash(hash);
      try {
        const pg = await PostgresFraudEntityRepository.findByHash(hash);
        compareObjects('FraudEntityRepository.findByHash', fs, pg);
      } catch (e) { logger.warn('dual_read_error', { error: e.message }); }
      return fs;
    }
    if (flags.ENABLE_FIRESTORE) return FirestoreFraudEntityRepository.findByHash(hash);
    if (flags.ENABLE_POSTGRES) return PostgresFraudEntityRepository.findByHash(hash);
    return FirestoreFraudEntityRepository.findByHash(hash);
  }
}
```

**Rationale for static methods (not DI framework):**
- 9-repository codebase is small enough that DI framework adds unjustified complexity
- Static methods are trivially testable via mock/stub at module level
- No runtime DI container needed; configuration is environment-driven
- If codebase grows beyond ~20 repositories, a DI framework (e.g., Awilix, Inversify) may be reconsidered

# ADR-003: Feature Flag Strategy

## Status
Approved

## Context
Decisions about which database provider is active must be evaluated dynamically by repositories, but hardcoding environmental checks across individual repository files violates centralization and risks inconsistent behavior.

## Decision
Introduce a central feature flag resolver `config/databaseProvider.js`. It will parse the `DATABASE_PROVIDER` environment variable, sanitize it against allowed values (`FIRESTORE`, `POSTGRES`, `DUAL_READ`, `DUAL_WRITE`, `SHADOW`), and export boolean helper flags (e.g., `isDualWrite()`, `isDualRead()`).

## Alternatives Considered
* **Distributed Checks:** Reading `process.env.DATABASE_PROVIDER` in every file. Rejected as it makes modifying flag logic or adding logging defaults prone to omissions.
* **Database/Redis-Stored Flags:** Storing flags in Redis or PostgreSQL. Rejected because fetching flags would require a database query, introducing latency and circular dependencies (we need the flag to know which DB to query).

## Consequences
* **Centralization:** All provider selection logic resides in one file.
* **Validation:** Invalid flag values default safely to `FIRESTORE`.

## Risks
* Typos in the environment variables could cause fallback behavior. This is mitigated by throwing errors or logging critical alerts on start if an unrecognized value is set.

## Rollback Implications
Updating the environment variable in Cloud Run or Vercel and performing a rolling restart changes the active database state in under a minute.

## Future Evolution
Once PostgreSQL is the permanent choice, the resolver can simply return static values before eventual removal.

# ADR-004: Dual Read / Dual Write Design

## Status
Approved

## Context
Running Firestore and PostgreSQL in parallel introduces challenges: transactional boundary differences, potential latency regression, and database-specific errors that could interrupt user operations.

## Decision
Design dual execution patterns in unified repository classes to protect users:
* **Dual Write (Non-blocking PG):** Writes are executed on Firestore first. If successful, Postgres is written in a `try/catch` block. If Postgres fails, the error is logged as a warning (`[MIGRATION_PARITY_DIFF]`) and the user's request completes successfully.
* **Dual Read (Comparison logging):** Reads are executed on Firestore and PostgreSQL. The results are compared using a helper class (`utils/dbComparer.js`). Any field mismatch is logged. The Firestore result is returned to the router/controller.

## Alternatives Considered
* **Strict Block Writes:** Failing the client request if either database write fails. Rejected because Postgres reliability is not yet verified at production load, and this would violate the "zero impact on production" rule.
* **Synchronous Multi-phase Commit:** Implementing a transaction coordinator (e.g., Two-Phase Commit). Rejected as overly complex and slow for our scale target of 10M users.

## Consequences
* **Client Protection:** Client requests remain fast and immune to Postgres failures.
* **Discrepancy Capture:** Drift is recorded immediately in logs.

## Risks
* Potential performance impact if Postgres writes are extremely slow. This is mitigated by timing writes and setting strict timeouts on PostgreSQL client pool queries.

## Rollback Implications
Requires no modifications to rollback strategy as Firestore remains the primary driver.

## Future Evolution
Provides the base comparison statistics for Phase 12 Data Drift evaluation.

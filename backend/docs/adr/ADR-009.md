# ADR-009: Granular Feature Flag System

## Status
Proposed (Generated during Phase 0 Architecture Audit)

## Context
Currently, all database routing is controlled by a single `DATABASE_PROVIDER` environment variable with 5 mutually exclusive values: `FIRESTORE`, `POSTGRES`, `DUAL_WRITE`, `DUAL_READ`, `SHADOW`.

The Production Migration Governance addendum requires **8 independent feature flags**:
1. `ENABLE_POSTGRES` — Whether PostgreSQL is available
2. `ENABLE_FIRESTORE` — Whether Firestore is available
3. `ENABLE_DUAL_WRITE` — Whether writes go to both databases
4. `ENABLE_DUAL_READ` — Whether reads compare both databases
5. `ENABLE_PARITY_LOGGING` — Whether parity differences are logged
6. `ENABLE_WRITE_VERIFICATION` — Whether write results are verified
7. `ENABLE_READ_VERIFICATION` — Whether read results are verified
8. `ENABLE_SHADOW_MODE` — Whether async shadow traffic is enabled

Additionally, all flags must support **runtime switching without recompilation**.

## Decision

**Implement a granular, runtime-configurable feature flag system.**

### Architecture

```javascript
// config/featureFlags.js
// Loads from env vars at startup; supports runtime hot-reload via admin endpoint
const flags = {
  ENABLE_FIRESTORE:       parseBool(process.env.ENABLE_FIRESTORE,       true),
  ENABLE_POSTGRES:        parseBool(process.env.ENABLE_POSTGRES,        true),
  ENABLE_DUAL_WRITE:      parseBool(process.env.ENABLE_DUAL_WRITE,      false),
  ENABLE_DUAL_READ:       parseBool(process.env.ENABLE_DUAL_READ,       false),
  ENABLE_PARITY_LOGGING:  parseBool(process.env.ENABLE_PARITY_LOGGING,  false),
  ENABLE_WRITE_VERIFY:    parseBool(process.env.ENABLE_WRITE_VERIFY,    false),
  ENABLE_READ_VERIFY:     parseBool(process.env.ENABLE_READ_VERIFY,     false),
  ENABLE_SHADOW_MODE:     parseBool(process.env.ENABLE_SHADOW_MODE,     false),
};

// Hot-reload: admin endpoint POST /api/v1/admin/feature-flags
// Updates in-memory flags without restart

// Backward compatibility: map DATABASE_PROVIDER to granular flags on startup
function mapLegacyProvider(provider) {
  const mapping = {
    FIRESTORE:  { ENABLE_FIRESTORE: true,  ENABLE_POSTGRES: false },
    POSTGRES:   { ENABLE_FIRESTORE: false, ENABLE_POSTGRES: true  },
    DUAL_WRITE: { ENABLE_FIRESTORE: true,  ENABLE_POSTGRES: true, ENABLE_DUAL_WRITE: true },
    DUAL_READ:  { ENABLE_FIRESTORE: true,  ENABLE_POSTGRES: true, ENABLE_DUAL_READ: true },
    SHADOW:     { ENABLE_FIRESTORE: true,  ENABLE_POSTGRES: true, ENABLE_SHADOW_MODE: true },
  };
  return mapping[provider] || mapping.FIRESTORE;
}
```

### Repository Decision Logic
```javascript
// Example in unified repository
if (flags.ENABLE_DUAL_WRITE) {
  await firestoreRepo.write(data);
  if (flags.ENABLE_PARITY_LOGGING) {
    try { await postgresRepo.write(data); } catch(e) { logParityDiff(e); }
  }
}
```

### Runtime Switching
- Admin endpoint: `POST /api/v1/admin/feature-flags` with JSON body
- Updates in-memory flag store atomically
- No server restart required
- Protected by admin role authorization
- Audit logged

## Alternatives Considered

**Option A: Keep current DATABASE_PROVIDER approach.**
- Simple but cannot independently toggle parity logging, verification, or shadow mode
- Cannot toggle flags at runtime
- **Rejected:** Does not satisfy governance requirements

**Option B: Use external config service (Consul, etcd).**
- Adds infrastructure dependency and latency
- **Rejected:** Over-engineering for 9-repo codebase

## Consequences
- **Flexibility:** Can independently toggle any feature flag
- **Runtime Control:** No restart needed for flag changes
- **Backward Compatible:** Maps existing DATABASE_PROVIDER to granular flags
- **Observability:** Flag changes are audit-logged
- **Overhead:** ~1 extra boolean check per repository operation (negligible)

## Risks
- Runtime flag changes could create inconsistent state if not atomic
- Mitigation: All flags read from a single in-memory object; update is synchronous



## Dual Write Strategy (Required by Architecture Review)

### Pattern: **"Write-Primary-First, Best-Effort-Secondary"**

```
DUAL_WRITE flow:
  1. Firestore write executes synchronously
  2. If Firestore succeeds:
     a. PostgreSQL write executes synchronously (try/catch)
     b. If PG succeeds → both databases consistent
     c. If PG fails → error logged as [MIGRATION_PARITY_DIFF]; FS result returned
  3. If Firestore fails → error thrown; PG never attempted
```

### Key Design Decisions:
- **Primary:** Firestore is ALWAYS the primary (per ADR-004, zero production impact rule)
- **Secondary:** PostgreSQL is ALWAYS best-effort (failure does NOT block user response)
- **No cross-database transaction** — accepted risk per ADR-004. The two databases have different transaction models (Firestore batch vs PG BEGIN/COMMIT). Two-phase commit is rejected for latency reasons at 10M user scale.
- **No event queue / outbox** — rejected because it adds infrastructure complexity disproportionate to a 9-repo codebase
- **No asynchronous writes** for dual-write — synchronous ensures error is caught immediately; async would require retry queue

## Rejected Patterns (Required by Architecture Review)

| Pattern | Rejected Because |
|---|---|
| **Outbox Pattern** | Requires separate outbox table + relay worker; over-engineered for 9 repos |
| **Event Queue (Kafka/RabbitMQ)** | Adds infrastructure dependency; not justified at current scale |
| **Two-Phase Commit** | Rejected per ADR-004: "overly complex and slow for 10M users" |
| **Async Dual Write** | Error would be deferred; no retry mechanism; harder to debug |
| **Change Data Capture (CDC)** | Requires PostgreSQL logical replication setup; out of scope for Sprint 2A |

## Flag Validation Rules (Required by Architecture Review)

Invalid combinations are rejected at startup with clear error messages:

| Combination | Valid? | Error Message |
|---|---|---|
| `DUAL_WRITE=true, FIRESTORE=false` | ❌ | "ENABLE_DUAL_WRITE requires ENABLE_FIRESTORE=true" |
| `DUAL_WRITE=true, POSTGRES=false` | ❌ | "ENABLE_DUAL_WRITE requires ENABLE_POSTGRES=true" |
| `DUAL_READ=true, DUAL_WRITE=false` | ❌ | "ENABLE_DUAL_READ requires ENABLE_DUAL_WRITE=true (cannot compare reads if not writing to both)" |
| `SHADOW=true, POSTGRES=false` | ❌ | "ENABLE_SHADOW_MODE requires ENABLE_POSTGRES=true" |
| `FIRESTORE=false, POSTGRES=false` | ❌ | "At least one database provider must be enabled" |

## Flag State Machine (Required by Architecture Review)

```
                 ┌──────────────────────────────────┐
                 │         FIRESTORE ONLY            │
                 │  FS=true, PG=false                │
                 └────────────┬─────────────────────┘
                              │ enable PG
                              ▼
                 ┌──────────────────────────────────┐
                 │         DUAL WRITE                │
                 │  FS=true, PG=true, DW=true        │◄──── SHADOW can run
                 │  Writes → FS+PG                   │      in parallel at
                 │  Reads  → FS only                 │      any stage after
                 └────────────┬─────────────────────┘      PG is enabled
                              │ enable DUAL_READ
                              ▼
                 ┌──────────────────────────────────┐
                 │         DUAL READ                 │
                 │  FS=true, PG=true, DR=true        │
                 │  Writes → FS+PG                   │
                 │  Reads  → FS+PG (compare)         │
                 └────────────┬─────────────────────┘
                              │ observation period
                              │ (14 days, 0% drift)
                              ▼
                 ┌──────────────────────────────────┐
                 │        POSTGRES ONLY              │
                 │  FS=false, PG=true                │
                 │  (Sprint 3 — after decommission)  │
                 └──────────────────────────────────┘

  SHADOW MODE (parallel, any stage after PG enabled):
  ┌──────────────────────────────────┐
  │  FS=sync, PG=async               │
  │  FS result returned to user      │
  │  PG result → metrics only        │
  └──────────────────────────────────┘
```

## Rollback Path

At any stage, immediate rollback:
- Set `ENABLE_FIRESTORE=true, ENABLE_POSTGRES=false`
- All reads/writes → Firestore only
- PG remains available for data recovery but not serving traffic

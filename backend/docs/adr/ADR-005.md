# ADR-005: Shadow Traffic Design

## Status
Approved

## Context
Validating PostgreSQL concurrent query execution, index performance, and pool capacity requires replaying real production traffic without impacting user request latency.

## Decision
Implement Shadow Traffic mode. When `DATABASE_PROVIDER=SHADOW`, the application behaves as follows:
1. Every query is executed on Firestore synchronously.
2. The user response is returned immediately.
3. An asynchronous worker execution block is scheduled (via `setImmediate` or fire-and-forget Promises) to run the equivalent query against PostgreSQL.
4. Latencies, query outcomes, and errors of the shadow query are caught and logged under `[SHADOW_TRAFFIC_LOG]`.

## Alternatives Considered
* **Synchronous Shadowing:** Running PostgreSQL queries in series. Rejected because it directly increases endpoint response times by the latency of the PostgreSQL execution.
* **Log Replay:** Recording requests in a log file and replaying them on a separate server. Rejected because it doesn't test PostgreSQL performance under the exact concurrent network conditions and connection pool limits of the production application server.

## Consequences
* **Performance safety:** API endpoints maintain Firestore speed.
* **Accuracy:** Captures true production read/write patterns on PostgreSQL.

## Risks
* Asynchronous calls could consume Node.js event-loop capacity if the task queue grows too fast. Mitigated by monitoring event-loop lag and limiting maximum active PostgreSQL connections in the pool.

## Rollback Implications
Has no impact on rollback stability.

## Future Evolution
Provides latency profiles and database connection telemetry for Phase 17 Operational Readiness checks.
